# BridgeORM Studio: UI/UX Specification

This document outlines the functional requirements and visual design for **BridgeORM Studio**, a high-performance graphical interface for managing databases powered by the BridgeORM Rust-Python core.

---

## 1. Design Aesthetics & Visual Language
- **Theme**: High-Contrast Dark Mode ("Submarine Grey" and "Electric Blue").
- **Typography**: Monospace for all data (Fira Code) to emphasize precision.
- **Feedback**: "Alive" UI with subtle gradients and micro-animations for FFI crossings.
- **Language Indicators**: 
    - **Blue Accent**: Represents Python-side operations (Asyncio Event Loop).
    - **Orange/Crab Accent**: Represents Rust-side execution (Tokio Runtime).
- **Core Branding**: "The Performance Bridge" – visual cues should emphasize the speed of the FFI boundary.

---

## 2. Feature Modules

### A. The Visual Schema Explorer (ERD & API)
*Instead of just text, developers should see their architecture and its exposure.*
- **Feature**: Auto-generated Entity Relationship Diagram.
- **Interaction**: Drag-and-drop to create relationships.
- **UI Logic**: When a user draws a line between two tables, the UI generates the Python `HasMany` or `BelongsToMany` code snippet in a side panel.
- **FFI Visualizer**: A small "Rust Bolt" icon next to each table name indicating it has been successfully whitelisted and cached in the Rust core (Deterministic Validation).
- **FastAPI Integrator**: A toggle that shows which models have auto-generated FastAPI routers and interactive documentation.

### B. Migration Timeline & Deterministic Snapshotting
*Managing the database lifecycle with confidence using a Source of Truth.*
- **Visual Diff**: A side-by-side comparison of `Model State` (Python Code) vs `Database State` (JSON Manifest).
- **Dry-Run Mode**: A "Simulate" button that runs the generated SQL migration in a temporary Rust transaction and reports success/failure without committing.
- **Timeline**: A vertical list of applied migrations with "Rollback to here" capabilities using the BridgeORM CLI backend.
- **Minimal DDL**: Highlighting the specific SQL DDL generated by the diffing engine to ensure transparency.

### C. The Telemetry Dashboard (Cross-Language Observability)
*Visualizing the observability bridge in real-time.*
- **Flamegraphs**: Visual representation of time spent in Python (serialization/deserialization) vs Rust (SQL execution/sqlx protocol).
- **FFI Boundary Metrics**: Real-time counter of language boundary crossings and "Vectorization Efficiency" (how many rows per crossing).
- **Slow Query Heatmap**: A map of the database schema where tables glow red if they are frequently involved in slow queries (>100ms).
- **Identity Map Hit Rate**: A gauge showing the efficiency of the task-scoped Identity Map (Task-level cache hits).

### D. The Query Laboratory (Generic Engine Tester)
*Testing the Generic Query Engine and Identity Consistency.*
- **Interface**: A dual-pane editor.
    - **Left**: Python Query Builder syntax (`User.query().filter(...)`).
    - **Right**: The raw SQL that the Rust engine generates in real-time.
- **Identity Tracker**: A visual marker that shows if a fetched object is a "New Instance" or a "Cache Hit" from the current `asyncio.Task` context variable.
- **Explain Plan**: An integrated "Explain" button that asks Rust to fetch the database's execution plan and visualizes it as a tree structure.

### E. Data Explorer (Zero-Copy View)
*Handling millions of rows with partial model enforcement.*
- **Infinite Scroll**: Utilizes the BridgeORM `fetch_lazy()` async iterator to stream data into the grid. The UI never freezes, even when browsing a table with 10 million rows.
- **Partial Select Enforcement**: Highlighting fields that were explicitly fetched vs. fields that are "Deferred" or "Ghosted" to prevent overfetching.
- **Security Audit**: A toggle that highlights any fields that are "Strictly Validated" by the Rust whitelister (Regex: `^[a-zA-Z_][a-zA-Z0-9_]*$`).

### F. Security & Safety Indicators
*Visualizing the "Security Mandate" of the bridge.*
- **SQL Injection Shield**: A real-time indicator that turns green when a query's identifiers pass the Rust Regex validator.
- **Panic Protection Status**: An indicator showing the health of the `catch_unwind` block on the FFI boundary, alerting if any Rust panics were successfully caught and translated to Python exceptions.
- **Strict Typing Badge**: A badge for models that have 100% type-hint coverage, ensuring full compatibility with the Rust engine's metadata parser.

## 3. UX Workflows

### Scenario 1: Fixing a Slow Query via Telemetry
1. The developer notices a "Slow Query" alert in the **Telemetry Dashboard**.
2. They click the alert, which opens the **Query Laboratory** with the offending SQL pre-loaded.
3. They use the **Explain Plan** feature to identify a missing index.
4. They switch to the **Schema Explorer**, add an `@index` decorator to the model, and hit **Generate Migration**.
5. The **Visual Diff** shows the new `CREATE INDEX` statement.
6. They hit **Apply**, and the Telemetry Dashboard immediately shows the query time dropping from 500ms to 2ms.

### Scenario 2: Verifying Identity Map Consistency
1. In the **Query Laboratory**, a developer runs two identical queries in a single "Simulated Task".
2. The **Identity Tracker** glows green for the second result, indicating it was retrieved from the per-task cache without a database roundtrip.
3. The **Telemetry Dashboard** confirms 0 Rust-side SQL calls for the second query.

### Scenario 3: Investigating a Blocked Dynamic Identifier
1. A developer attempts to use a dynamic table name from a user-input field in the **Query Laboratory**.
2. The **SQL Injection Shield** immediately turns red, highlighting that the identifier failed the Rust Regex validation.
3. The UI explains *why* the identifier was rejected and provides the correct pattern (`^[a-zA-Z_][a-zA-Z0-9_]*$`).
4. The developer sanitizes the input, the shield turns green, and the query is allowed to cross the FFI boundary.

## 4. Technical Implementation
- **Frontend**: React + TypeScript + TailwindCSS for the main UI.
- **Visualizations**: D3.js or X6 for the ERD; Recharts for telemetry.
- **Backend Bridge**: A specialized FastAPI server that imports the `bridge_orm` Python package and exposes its metadata via WebSockets for real-time telemetry.
- **State Synchronization**: Uses `pyo3-async-runtimes` to pipe Tokio-level performance metrics directly to the WebSocket stream.
- **Context Handling**: Integration with Python's `contextvars` to simulate task-scoped behavior in the Query Laboratory.
